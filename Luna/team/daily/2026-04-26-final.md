## 🔑 Key Architecture Decisions

1. **Single business** — `companyProfile/main` singleton
2. **Customer snapshots** ฝังในทุก tax-bound document → ที่อยู่/TIN frozen ตอนออก (ตามกฎหมายภาษี)
3. **Document numbering** ใช้ Firestore transaction → กันชนกันใน concurrent write
4. **Tax-bound documents immutable** ใน Firestore rules (`delete: false`) → ใช้ cancellation + credit note workflow
5. **Audit log** เป็น `arrayUnion` field ใน document → simple lookup, no separate collection
6. **PromptPay QR** generate ตอน save (`qrCodeData` field) → render เป็น PNG ตอน download PDF
7. **VAT inclusive/exclusive toggle** ทั้ง Quotation/Invoice — รองรับธุรกิจ B2C ที่ลงราคารวม VAT
8. **WHT support** ทั้งฝั่งรับ (invoice payment) + ฝั่งจ่าย (expense) → ครบกฎหมาย ภงด.3/53

## ⚠️ Known Limitations / Manual Steps After Go-Live

1. **ไม่มี e-Tax Invoice** — ตามที่ตกลงกับพี่แมน เฟสนี้ไม่ทำ
2. **VAT รายเดือน** — UI พร้อม แต่ accountant ต้องเข้าระบบสรรพากรเอง (auto-submit ทำไม่ได้ — RD ไม่มี API)
3. **PND.3/53 PDF official format** — มี CSV export แล้ว, official PDF (สำหรับ download/print) ทำเฟสถัดไป
4. **Inventory ⇄ Production COGS link** — schema พร้อม (`isCOGS` + `relatedOrderId`), UI workflow ค่อยเพิ่ม
5. **Migration script** ต้องรันก่อน go-live (`node scripts/migrate-customers.mjs`)
6. **Resend + LINE env vars** ต้องตั้งใน hosting (`RESEND_API_KEY`, `LINE_CHANNEL_ACCESS_TOKEN`)
